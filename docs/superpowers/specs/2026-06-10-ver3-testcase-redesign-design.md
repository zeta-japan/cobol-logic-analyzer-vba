# ver3.0 テストケース再設計 — 設計仕様

日付: 2026-06-10 / 状態: 承認済 (設計レビュー完了)

## 目的

単体テストの「全分岐疏通」(C1 分岐網羅) を支援する。テスターが

1. ケース数を即座に把握できる
2. 各ケースが通る検証Point (分岐アーム) と実行文を把握できる
3. ケース毎の入力設定値と想定結果を得られる
4. Driver 雛形でそのまま投入できる

状態にすることがゴール。

## 用語定義

| 用語 | 定義 |
|---|---|
| 検証Point | 分岐アーム。IF の THEN / ELSE (ELSE 省略時も「条件不成立」アームを数える)、EVALUATE の各 WHEN + WHEN OTHER (省略時は skip アーム)、SEARCH の AT-END / 各 WHEN。複合条件 (AND/OR) は 1 分岐として扱い直交積はとらない |
| テストケース (シナリオ) | PROCEDURE 入口から終了点まで貫通する実行パス 1 本 |
| 終了点 | GOBACK / STOP RUN / EXIT PROGRAM (自動検出) + 「終了扱いセクション」(コントロールシートで登録。ABEND 処理段など) |
| 最終Action | 終了点直前に実行される最後のアクション (= プログラムが本当に終わる箇所) |

## ケース生成規則

1. **実行流パス枚挙**: PROCEDURE 入口セクションから PERFORM 呼出/復帰を辿り、終了点で停止する。
1-2. **実行不能パスの排除 (リテラル定数伝播)**: パスに沿って「リテラルを代入された項目」を追跡し (`MOVE 'リテラル' TO 項目`)、CALL の USING 引数・演算/STRING/INITIALIZE の代入先は「不明」に失効させる。`項目 = 'リテラル'` (同一項目の OR 連結含む) / `項目 NOT = リテラル` 型の分岐条件で項目値が既知の場合、矛盾するアームを枚挙から除外する (例: FETCH 該当有 → 結果フラグ '0000' なのに '0001' 側アームを通るパス)。**確実な場合のみ剪定** (値不明なら両アーム可)。副作用として、恒真/恒偽の分岐はマトリクス上「被覆不能行 (赤)」として現れる = デッドコード検出。
2. **正常系** = 終了扱いセクション/合成異常を経ずに GOBACK へ到達するパス。**C1 貪欲法**で「全アームを最少本数で被覆」する代表パス集合を選ぶ。両アームが直後に合流する非分岐性アーム (例: 日付による年度補正 IF) は新ケースを増やさず、既存ケースの**前提条件として分配**する (貪欲法が自然にこの形になる)。
3. **異常系 (コード由来)** = 終了扱いセクションへ到達する分岐アームごとに 1 ケース (最短前置パス)。
4. **異常系 (合成)** = 外部 CALL ごとに「呼出先異常 → 終了」のスタブケースを 1 件生成する (終了扱いセクション内の CALL は除外)。呼出先が内部で異常終了する運用慣行に対応。不要なものはテスターが削除する。
5. **並び**: 正常系 → 異常系。異常系は「今回対象外・機上確認推奨」標記とし、入出力-想定結果 / Driver 雛形の生成対象外。

## 出力シート

### 1. テストケース候補 (改造)
ケース毎の縦ブロック (ステップフロー形式):
- 処理段ラベル (SECTION 直前コメントの「（…）」を優先、無ければ SECTION 名)
- 外部交互 (CALL / EXEC) とその結果アーム (例: `EXEC FETCH → ADACODE=3`)
- キー代入 (後続の分岐条件に影響する項目への代入。例: `W-RESULT-CD ← '0001'`)
- 終了形態 (GOBACK 正常 / 終了扱いセクション経由の異常終了 / CALL 先異常)

### 2. 分岐カバレッジ表 (新規・主視図)
- 行 = 検証Point (分岐アーム、開始行ハイパーリンク付き)
- 列 = ケース、✓ = 該当パスがそのアームを通る
- **全空行 = 漏れ → 赤ハイライト**
- 下部に 終了形態 / 系 (正常・異常) / 対象 (テスト対象・機上確認) の行

### 3. ロジック階層(実行順展開)
ケース標記 (決定パス / 最終Action / 実行行着色、前へ・次へボタン) を新ケース TC1..TCn で駆動する。UI は従来どおり。
PERFORM 先をインライン展開する位置には、展開本体と字下げを揃えた**セクションバナー行**を挿入する (展開境界の視認性向上):
`■--------------- 名前 SECTION ---------------` (太字、開始行ハイパーリンク付き) **← 実装済 (2026-06-10)**

### 4. 入出力-想定結果 (旧「入力項目」を置換)
正常系ケース毎に:
- **入力**: ① LINKAGE 入力項目の設定値 ② DB 前提 (該当レコード有無など) ③ サブプログラム戻り値前提
- **出力想定値**: パス上の最終代入 (last-write-wins)。リテラル = 値、単純移送 = `=入力XXX`、サブ戻り値 / DB 由来 / 演算 = 式表示

### 5. Driver雛形 (旧「Driver_Dummy雛形」を置換)
正常系ケース毎に `MOVE 入力値 → CALL 対象 → DISPLAY 出力項目`。DB / サブ前提はコメントで明記。**Dummy 生成は廃止**。

## エンジン拡張 (前提工事)

1. **文のマージ** (正規化フェーズ):
   - 行末が OR / AND → 次行を連結 (複数行条件)
   - `EXEC` 〜 `END-EXEC` を 1 文に連結
   - `STRING` 文: DELIMITED / INTO 行を連結
   - `CALL … USING` の被演算子のみの後続行 (キーワード除外) をピリオド到達行まで連結
   - マージ後の文の行番号 = 先頭行
2. **動詞追加**: EXEC (例: `EXEC ADABAS OPEN K77`)、STRING (INTO 先を出力代入として抽出)、INITIALIZE、ACCEPT
3. **コメント捕捉**: SECTION 直前のコメント行を処理段ラベルとして保持
4. ADACODE 等のレジスタ項目は「外部要因」として扱う (入力前提に分類)

## 設定 (コントロールシート)

「終了扱いセクション」入力欄 (複数登録可)。空の場合は早期終了なし (従来挙動)。ワークブックに保存され解析時に読み込む。

## 受け入れ基準 (fixture ICASE3)

カーソル OPEN / FETCH / CLOSE + 日付サブ呼出 + 業務サブ呼出 + ABEND 段、という典型形の fixture で:

- 検証Point = **14** (7 分岐 × 2 アーム)
- 実行可能パス = 正常 **6** 本 (構文上 12 本 → 定数伝播で矛盾パス 6 本を排除)
- ケース = **正常 3 + 異常 5** (コード由来 3 + 合成 2)。内訳:
  - TC1 = OPEN正常 → 日付(1〜3月) → FETCH 該当無(='0001') → CLOSE正常 → サブ呼出 戻り='1'
  - TC2 = OPEN正常 → 日付(4〜12月) → FETCH 該当有(='0000') → CLOSE正常 → 終了
  - TC3 = TC1 系で サブ戻り≠'1' (出力='9999')
  - TC4〜TC8 = 合成(日付サブ異常/業務サブ異常) + コード由来(OPEN/FETCH/CLOSE異常)、トリガー行順
- 分岐カバレッジ表が全行被覆 (漏れ 0)
- テストケース候補がステップフロー形式で出力される
- 入出力-想定結果と Driver 雛形が正常系 3 件分生成される

(上記は PS オラクル `ver3-oracle` で 2026-06-10 全アサーション通過を確認済)
